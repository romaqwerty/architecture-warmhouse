### 1. Описание функциональности монолитного приложения
Система обладает следующим функционалом:

**Управление отоплением:**
- Пользователи могут:
  - Регистрировать датчик с указанием статуса (активный/неактивный), типа, имени, и места расположения (комнаты)
  - Обновлять метаинформацию датчика с указанием статуса (активный/неактивный), типа, имени, и места расположения (комнаты)
  - Удалять датчик 
  - Получать метаинформацию одного датчика из БД POSTGRES
  - Получать метаинформацию нескольких датчиков из БД POSTGRES
- Система поддерживает
  - Возможность получение метаинформацию датчиков пользователем от системы по REST API
  - Хранение последнего события регистрации\обновления метаданных датчика в БД POSTGRES


**Мониторинг температуры:**

- Пользователи могут
  - Получать температуру с одного датчика по его ID
  - Получать температуру со всех датчиков системы
  - Получать температуру c одного датчика по названию комнаты (location)
  - Изменять значения температуры датчика по его ID и сохранять его в БД 
- Система поддерживает
  - загрузку значения температуры от датчика
  - получение значения температуры по ИД датчика во внешнем сервисе по REST API
  - получение значения температуры по названию комнаты (location) во внешнем сервисе по REST API
  - Хранение последнего события измерения температуры в БД POSTGRES

### 2. Анализ архитектуры монолитного приложения

- Система спроектирована на основе клиент-серверной архитектуры. Архитектурный стиль монолит - все бизнес-логика 
  сосредоточена в одном модуле, единице развертывания.
- Система делает синхронный HTTP вызов для получения температуры по ID датчика в другой сервис
- Система написана на языке GO
- Система использует БД POSTGRES
- Используется HTTP REST API с синхронной семантикой (клиент блокируется в ожидании) для коммуникации между пользователями и системой


### 3. Определение доменов и границы контекстов

  1. Домен управление климатом
    1.1 Поддомен управления температурой
       1.1.1 Контекст измерения температуры
    1.2 Поддомен управления устройствами
       1.2.1 Контекст создание/обновление/удаления устройств
       1.2.2 Контекст изменения состояния устройств



### **4. Проблемы монолитного решения**
- Отсутствие архитектурных границ (нет явного разделения на контексты)
- Отсутствие агрегатных границ. В запросах приходят данные сразу по температуре, датчиках, отоплению. Возникает путаница
- Контексты шарят одну схему данных в бд
- Плохо масштабируется при росте количества датчиков (синхронные запросы занимают ресурсы сервиса)
- Читающая и пишущая нагрузка идет на один инстанс БД - рост чтений может вызвать недоступность бд на запись
- Жесткая связность между модулями - прямой ХТТП вызов сервиса температуры.
- Обновление требует остановки приложения

# Задание 5. Работа с docker и docker-compose

Перейдите в apps.

Там находится приложение-монолит для работы с датчиками температуры. В README.md описано как запустить решение.

Вам нужно:

1) сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе /temperature?location= будет отдавать рандомное значение температуры.

Locations - название комнаты, sensorId - идентификатор названия комнаты

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
```

2) Приложение следует упаковать в Docker и добавить в docker-compose. Порт по умолчанию должен быть 8081

3) Кроме того для smart_home приложения требуется база данных - добавьте в docker-compose файл настройки для запуска postgres с указанием скрипта инициализации ./smart_home/init.sql

Для проверки можно использовать Postman коллекцию smarthome-api.postman_collection.json и вызвать:

- Create Sensor
- Get All Sensors

Должно при каждом вызове отображаться разное значение температуры

Ревьюер будет проверять точно так же.
